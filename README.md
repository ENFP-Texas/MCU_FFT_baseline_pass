# MCU 8-point FFT Baseline

本仓库保存数字电路与微处理器系统设计课程设计中，使用自定义 MCU 完成 8 点 FFT 运算的 baseline 与低风险速度优化版本。

当前版本已经从最初的功能正确 baseline，优化到：

* `cnt_test = 0x00a0 = 160 cycles`
* MCU 工作频率提升到 `95 MHz`
* 95 MHz bitstream 已生成并上板验证
* ILA 中观察到 `cnt_test = 000a0`，`done` 正常拉高，`verify_we` 正常写入
* FFT 输出仍通过 `verify_RAM / verify_vector_out` 输出
* 优化方式仍保持 MCU 指令执行 FFT，没有改成专用 FFT 硬件加速器

## 速度优化过程

本项目目前完成的是低风险速度优化，没有大幅修改 MCU 微架构。

### Baseline

初始 baseline 版本使用 MCU 执行 8 点 FFT，功能正确，上板 ILA 和 Python 脚本均验证通过。

初始运行周期大约为：

```text
cnt_test ≈ 224 cycles
```

---

### Stage 1：输入搬运优化

原始程序先把 `test_ROM` 中的输入搬运到内部 RAM，然后再从内部 RAM 读取进行 FFT。

优化后，Stage 1 直接从 `test_ROM` 读取输入数据参与计算，删除输入搬运指令。

结果：

```text
cnt_test ≈ 192 cycles
```

---

### Stage 2：最终输出直写 verify_RAM

原始程序在 FFT 结束后，先把最终结果写回内部 RAM，然后再从内部 RAM 读出并写入 `verify_RAM`。

优化后，最后一级 FFT 直接把结果写入 `verify_RAM`，并按照原始输出映射保持输出顺序正确。

结果：

```text
cnt_test = 0x00a0 = 160 cycles
```

最终输出顺序保持为：

```text
0, 1, 8, 9, 4, 5, 12, 13, 2, 3, 10, 11, 6, 7, 14, 15
```

---

### Stage 3：提高工作频率

在保持 `cnt_test = 160 cycles` 不变的基础上，使用 Clocking Wizard 从板载 50 MHz 输入时钟生成更高频 MCU 时钟。

当前低风险测试过程包括：

| 版本       |   时钟频率 | 说明                                          |
| -------- | -----: | ------------------------------------------- |
| speed_v2 | 50 MHz | `cnt_test = 160`                            |
| speed_v3 | 75 MHz | timing 通过，上板验证                              |
| speed_v4 | 84 MHz | timing 通过                                   |
| speed_v5 | 95 MHz | bitstream 生成完成，上板 ILA 验证 `cnt_test = 000a0` |

当前推荐稳定版本：

```text
speed_v5_clk95
```

对应 tag：

```text
speed_v5_cnt160_95MHz
```

## 为什么不同输入数据下速度基本不变？

当前 FFT8 程序是完全展开的 straight-line ASM：

* 没有循环
* 没有数据相关分支
* 没有根据输入数据提前结束
* 没有根据输入数据选择不同路径

因此，只要老师仍然测试的是一组 8-point FFT 输入，程序执行的指令数量固定，`cnt_test` 也固定。

输入数据会改变 FFT 输出结果，但不会改变执行周期数。

当前速度公式为：

```text
T_exec = 160 / f_clk
```

例如：

```text
95 MHz: T_exec ≈ 1.684 us
```

## 关键文件说明

### Verilog / RTL

* `src/board_top.v`
  顶层模块。当前版本接入 `clk_wiz_0`，使用 95 MHz `clk_mcu` 驱动 MCU、test ROM、verify RAM 和 ILA。

* `src/mcu_top.v`
  MCU 顶层模块，连接控制单元、寄存器堆、ALU、指令 ROM、数据 RAM 和外部测试/验证接口。

* `src/mcu_core.v`
  MCU 核心执行逻辑。

* `src/alu.v`
  ALU，包括 FFT 运算中使用的加减乘等操作。

* `src/test_rom_ip.v`
  test ROM 包装模块，用于读取测试输入向量。

* `src/verify_ram_ip.v`
  verify RAM 包装模块，用于保存 FFT 输出。

---

### ASM / 指令文件

* `asm/fft8_speed_v1.asm`
  第一阶段速度优化版本，减少输入搬运。

* `asm/fft8_speed_v2.asm`
  第二阶段速度优化版本，最终 FFT 输出直接写入 `verify_RAM`。

* `mem/instr_fft8.mem`
  当前用于 Vivado 指令 ROM 初始化的机器码文件。

* `mem/instr_fft8.coe`
  Vivado IP 可使用的 COE 格式指令初始化文件。

ASM 修改后应通过 assembler 重新生成 `.mem` 和 `.coe`，不建议直接手改机器码。

---

### IP 文件

* `ip/test_rom_dist/test_rom_dist.xci`
  Distributed Memory Generator ROM IP 配置。

* `ip/clk_wiz_0/clk_wiz_0.xci`
  Clocking Wizard 配置文件。当前用于从 50 MHz 输入生成 95 MHz MCU 时钟。

## Vivado 使用方法

如果从仓库重新打开工程，需要确认以下内容：

1. `board_top.v` 使用 `clk_mcu`，而不是直接使用 `CLK_50M`。
2. `clk_wiz_0` 输入为 50 MHz，输出为当前目标频率，例如 95 MHz。
3. `test_ROM`、`mcu_top`、`verify_RAM` 使用同一个 `clk_mcu`。
4. ILA/debug clock 也应使用 `clk_mcu` 或 `clk_wiz_0/clk_out1`。
5. 修改 `.mem`、`.coe` 或 Clocking Wizard 后，需要重新：

   * Run Synthesis
   * Run Implementation
   * Generate Bitstream
   * Program Device

## 现场更换随机测试数据的方法

如果老师提供新的测试输入：

1. 修改 `mem/test_vector.coe` 或对应 test ROM 初始化文件。
2. 重新生成 `test_rom_dist` IP Output Products。
3. 重新运行：

   * Run Synthesis
   * Run Implementation
   * Generate Bitstream
4. Program Device。
5. 用 ILA 检查：

   * `test_vector_in`
   * `verify_vector_out`
   * `cnt_test`
6. 用 Python 脚本检查结果是否 PASS。

只要仍然是 8-point FFT，且程序不变，`cnt_test` 应保持为：

```text
0x00a0 = 160 cycles
```

## 当前版本用途

当前版本作为低风险速度优化后的 MCU FFT baseline：

* 用于速度榜的稳定提交版本
* 用于后续更激进优化前的备份
* 用于继续尝试更高频率、更少指令数或新增 MCU 指令
* 用于硬件效率榜分析时的核心版本参考

后续可能优化方向：

* 继续提高工作频率，但需要保持 WNS 为正
* 去除 ILA/debug 版本以改善 WNS 和资源
* 针对 FFT 增加更高效的 MCU 指令
* 优化 ALU 关键路径
* 减少不必要的 debug / LED 逻辑
* 进一步压缩 FFT ASM 指令数量
